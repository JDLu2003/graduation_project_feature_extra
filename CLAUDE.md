# 项目指导准则：多模态对话特征提取 (Graduation Project Sub-module)

## 1. 核心任务与范围
- **核心目标**：针对验证集（及后续训练集）数据进行多模态（文本、视频、情绪标签）特征提取工作。
- **任务边界**：**仅**专注于特征提取与特征持久化保存，绝对不需要关注或编写后续的分析和建模代码。
- **配置驱动**：所有策略、路径、维度等魔法数字必须通过配置文档或配置类（如 `config.yaml` 或专用的配置模块）进行灵活管理，不可在代码中硬编码。

## 2. 环境与技术栈
- **编程语言**：Python 3.x（**必须极其严格地使用 Type Hints 类型标记**）。
- **核心框架**：PyTorch。
- **环境管理**：使用 Conda 维护，环境名称为 `security`。激活命令：`conda activate security`。

## 3. 数据输入说明 (Input Specification)

数据集位于当前工作目录的上一级文件夹 `../data_set/` 下。当前验证集的核心处理路径为 `../data_set/Viedeo_en_dev/`。

### 3.1 目录结构与示例
输入数据的层级结构如下所示：
```text
../data_set/
└── Viedeo_en_dev/
    ├── dev.txt                # 包含对话结构与情绪标记的文本文件
    └── Video_en_dev/          # 视频文件根目录
        ├── C_18/              # 对应某一组对话 (如对话 18)
        │   ├── C_18_U_1.mp4   # 对话中的第 1 句话
        │   └── ...
        └── C_1001/
            ├── C_1001_U_1.mp4 # C_{对话编号}_U_{句子编号}.mp4
            └── ...

```

### 3.2 数据对应与解析逻辑

程序需要解析 `dev.txt` 并与视频文件进行强绑定。`dev.txt` 的解析规则必须严格遵循：

1. **对话头 (Dialogue Header)**：以两个数字开头的行，例如 `18 13`。代表后续紧跟的数据属于**对话 18**，且该对话总共包含 **13 句话**。对应的视频文件夹必须映射为 `Video_en_dev/C_18/`。
2. **语句行 (Utterance Line)**：对话头之后紧跟的具体对话内容，使用 `|` 分割字段。
* **格式示例**：`1 | Alright . Phoebe ? | Monica | neutral (null) | Chandler | joy (1) | Phoebe | joy (1)`
* **字段解析语义**：`句子编号 | 文本内容 | 说话人名字 | 说话人情绪 | 听众1名字 | 听众1情绪 | 听众2名字 | 听众2情绪 | ...`
* **视频映射**：该行对应视频文件为 `Video_en_dev/C_18/C_18_U_1.mp4`。


3. **人员构成约束**：每一句话**一定有且只有一个说话人**，但可能有 $0$ 到 $N$ 个非说话人（听众）。

## 4. 数据输出规范 (Output Specification)

### 4.1 输出目录结构 (镜像构建)

特征输出目标文件夹统一定义为 `../feat_out/`。
为了保证数据的易读性和管理，**输出文件夹内部必须严格镜像输入视频的文件夹结构**，自动创建缺失的父级目录。

* **输入示例**：`../data_set/Viedeo_en_dev/Video_en_dev/C_1001/C_1001_U_1.mp4`
* **输出示例**：`../feat_out/Video_en_dev/C_1001/C_1001_U_1.pt`

### 4.2 Tensor 特征维度与格式要求

* **提取目标**：针对每一句话的每一个人（包含说话人和非说话人），提取出一个 1024 维的特征向量。
* **保存格式**：使用恰当的方式保存为 PyTorch 的 `.pt` 文件。
* **Tensor 维度**：必定为 `[N, 1024]`，其中 N 代表该句话中出现的总人数（说话人 + 非说话人）。
* **Tensor 排序（核心依赖）**：Tensor 的第 0 维必须严格按照 `dev.txt` 中人物出现的顺序排列。**索引 0 始终为说话人**，其后依次为解析出的非说话人。
* **缺失画面处理策略**：由于视频片段通常只包含说话人的画面，若采用视觉模型，需明确定义非说话人特征的填充策略（如：使用全零向量补齐、使用文本/情绪标签生成特征、或通过上下文视频推断等），并将此策略记录在配置或说明文档中。

## 5. 软件工程与框架设计要求

* **目录结构后置**：当前阶段不提前规划死板的代码文件夹结构，应根据实际业务逻辑的复杂度和软件工程要求，在开发过程中合理化、规范化地进行模块拆分和明确。
* **开闭原则 (OCP) 与面向对象**：必须设计合理的框架和统一的特征提取接口（如抽象基类）。当后续需要引入新的提取方法（纯文本预训练、视觉模型、多模态融合等）时，只能通过实现特定接口来扩展，**绝对不需要修改现有的核心流程代码**。
* **防御性编程 (Asserts)**：在输入-处理-输出的全流程中，应当大量使用 `assert` 来确保数据一致性。必须验证：
* 预期读取的视频文件或路径真实存在。
* `dev.txt` 解析出的字段数量与逻辑合法。
* 最终生成的 Tensor 第一维（N）与该行解析出的人物总数完全一致，且第二维必定为 1024。


* **最佳实践**：应当使用深度学习科研领域和软件工程领域的最佳实践来编写代码，确保代码的质量、性能、结构清晰且易于维护。

## 6. 开发流程与 Git 规范

必须使用 Git 对代码进行有效的本地版本控制，确保开发过程透明可追踪。

### 6.1 Git 提交规范 (Conventional Commits)

要求每个功能模块提交规范的 Commit Message：

* `feat:` 新增功能（如新增解析器、新增某种特征提取策略）
* `fix:` 修复 Bug（如修复路径拼接错误、维度不匹配）

* `docs:` 文档更新（如更新 CLAUDE.md 或开发进度）
* `chore:` 杂项（环境配置、依赖安装等）

### 6.2 进度记录与持久化

* 需要明确记录开发进度。记录每个阶段的进展情况，包括完成的任务、遇到的问题和解决方案等。

* 开发进度记录在`PROGRESS.md`中，按照日期和功能模块进行详细记录。
